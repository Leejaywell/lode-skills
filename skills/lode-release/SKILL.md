# Release Builder（发布版本）

主线第 ⑥ 环。收尾环节：**隐私审计 → 打包 → 发布**，产出可交付给用户的版本。

## Usage（什么时候用）

- 所有计划内切片已完成且通过 `lode-review` 审查。
- 准备出可分发的产物（如 Mac/Electron 包、可部署构建）。
- 发版前需要做隐私/安全的最后一道关。

## 团队 / 长生命周期：PR/CI 完成模式

solo 模式下"完成"= 本地构建 + `review-passed`。**团队 / 长生命周期项目切到 VCS 原生完成**：
- 每个切片/Epic 开**分支**，原子提交，开 **PR**。
- 子代理审查（`lode-review`）作为 **PR 前过滤器**先跑一遍；
- "完成" = **PR 过 CI + 必需 approval → 合并入主干**，不是本地标记。
- 合并后才更新账本为 passed；冲突由主 Agent 解。
- 这把 self-review 升级成 peer review，正面支持多人协作。

## Done（什么算合格）

- **隐私审计通过**：无硬编码密钥/令牌，无意外收集/上传用户数据，权限申请最小且有据。
- 生产构建成功，按目标平台打出可运行产物。
- 写出发布说明（版本号、变更摘要、已知限制），来源于 `changelog.md`。
- 给出验收/安装方式，用户能据此跑起来。
- **主动演示能跑**：打完包/构建后亲自跑一下并截图，或给一条「这样装、这样跑」的命令，让用户当场看到成品在跑——不只交一个包。
- **团队模式**：所有 PR 已过 CI + 审批合并；发布从主干打。

## Guardrails（红线）

- 审查没过、切片没全完成 → 不允许发布（受门禁约束）。
- 隐私审计发现 CRITICAL 问题 → 立即停下修复，不带病发版。
- 不在发布环偷偷加新功能；只做收口与打包。
- 发版动作（推送/上架等对外不可逆操作）执行前先向用户确认。
