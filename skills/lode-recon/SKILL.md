# Recon（代码侦察）

棕地入口环（绿地项目跳过）。在 spec/plan **之前**先把现有系统摸清，产出一份 `system-map.md`，让后续的 spec 知道"现状是什么"、plan 知道"动哪会爆"。

> 第一性原理：不懂现有系统就规划，等于蒙着眼睛改别人的代码。先看清，再动手。

## Usage（什么时候用）

- 目标落在**已有项目**（有代码、有历史），不是从零起。
- 要改造/扩展/迁移现有功能，且**不能破坏现状**。
- 进入 `lode-spec`（delta 模式）/ `lode-plan` 之前的第一步。

## 怎么侦察（看清，别猜）

优先用结构化工具而非纯靠读：有 codegraph/LSP 就用它查"谁调用谁、改这会牵动什么"；否则用 grep + 关键文件通读。重点摸清：

1. **架构与边界**：模块/分层、入口、数据流、外部依赖与集成点。
2. **约定**：命名、目录组织、错误处理、配置、既有代码风格（后续要"长得像它"）。
3. **怎么跑**：构建 / 测试 / 运行 / lint 的**真实命令**（写进 verify.sh 的素材）。
4. **测试基线**：现有测试**当前是否全绿**、覆盖到哪、哪些区域没测（改这些区域风险高）。
5. **热点与风险**：高耦合、巨文件、无测试的核心路径、安全/数据敏感面。

## Done（什么算合格）

产出 `.lode/<project>/system-map.md`（起步模板见 `docs/templates/system-map.md`），且满足：
- 架构地图：模块/分层 + 关键入口 + 数据流，能让人据此定位代码。
- 约定清单：命名/目录/错误处理/配置/风格——后续改动据此"长得像现有代码"。
- **可跑命令**：构建、测试、运行、lint 的真实命令（直接喂给 `verify.sh`）。
- **基线快照**：跑一遍现有测试，记下当前绿/红状态与覆盖盲区——这是日后判断"我弄坏了什么"的参照系。
- 风险区：高耦合/无测试/安全敏感的位置点名，供 plan 做影响分析。

## Guardrails（红线）

- **只看不改**：侦察阶段不写业务代码、不重构；发现要改的记进地图，留给 plan/build。
- 用证据说话：架构结论来自实际调用关系/代码，不靠对仓库名的想象。
- 基线必须**实跑**一次测试，不能假设"应该是绿的"。
- 地图服务于"安全改动"，不求面面俱到——聚焦将要触碰的区域，别给整库写百科。
