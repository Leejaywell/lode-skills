# Dev Planner（开发计划）

主线第 ④ 环。把 `Product-Spec.md`（及设计产物）拆成一组 **Face**。

> **Face** = 一个**纵向可验收切片**：做完它能编译、能跑、能独立验收，而不是横向的"先写完所有 model 再写所有 UI"。

## Usage（什么时候用）

- `Product-Spec.md`（可选 `Design-Brief.md` / `mockups/`）已确认，准备开工。
- 需要把"要做什么"变成"按什么顺序、分几片做完"。
- 进入 `lode-build` 之前，必须先有计划。

## Done（什么算合格）

产出 `.lode/<project>/DEV-PLAN.md`，且满足（含当前文档状态说明、技术选型结论、Face 规划与先后/可并行标注、必要的数据库设计与开发规则）：
- 拆成有序的 Face 列表，每个 Face 都是能跑起来的纵向切片。
- **每个 Face 自带一条 Goal**：目标 / 完成标准（可程序判断）/ 验收方式。
- 标出 Face 间依赖；互不依赖的标记为**可并行**（供主 Agent 决定是否拆子 Agent）。
- 技术选型与关键架构决策有简短说明和理由。
- 第一个 Face 是"最薄可跑通"的骨架，能尽早验证范式跑得通。
- 定义本项目的**确定性验证命令**（编译 + 测试），供 lode-build 落成 `.lode/<project>/verify.sh`、由 Stop 门禁实跑——把"编译/测试过没过"从模型自评移交给程序。

## Guardrails（红线）

- 不要拆成一堆碎工具/碎函数任务 —— 给能力、给可验收切片，让 Builder 自己组织实现。
- 每个 Face 的完成标准必须可程序判断，不能是"差不多"。
- 计划服务于循环，不追求一次写死；允许 Build 过程中回来调整。
- 若已有设计原型代码，计划要明确**在其基础上直接开发、复用代码**，而非"仅作参考"重写。
- 计划先给用户拍板再进 `lode-build`。
