# Dev Planner（开发计划）

主线第 ④ 环。把 `docs/spec.md`（及设计产物）拆成一组 **切片**。

> **切片** = 一个**纵向可验收切片**：做完它能编译、能跑、能独立验收，而不是横向的"先写完所有 model 再写所有 UI"。

## Usage（什么时候用）

- `docs/spec.md`（可选最新 `.lode/design/` / `mockups/`）已确认，准备开工。
- 需要把"要做什么"变成"按什么顺序、分几片做完"。
- 进入 `lode-build` 之前，必须先有计划。

## Done（什么算合格）

产出开发计划，落地到 `.lode/plan/<功能简称>-<YYYY-MM-DD_HH_MM_SS>.md`（**每次 replan 生成新文件、绝不覆盖旧版**，保留计划演进史；时间戳取到秒防撞名。取最新：`ls -t .lode/plan/*.md | head -1`）。内容满足（含当前文档状态说明、技术选型结论、切片规划与先后/可并行标注、必要的数据库设计与开发规则）：
- 拆成有序的切片列表，每个切片都是能跑起来的纵向切片。
- **每个切片自带一条 Goal**：目标 / 完成标准（可程序判断）/ 验收方式 / **验收场景**。
- **验收场景必须先于开发定**（spec-bound）：从 spec 的验收标准派生出几条具体、可执行的场景（"给定 X，做 Y，得到 Z"），它们规定**测试要测什么**。这样测试绑的是**需求**，不是 builder 写完代码再反推的弱测试——堵住"测试绿但功能跑偏"。
- 标出切片间依赖；互不依赖的标记为**可并行**（供主 Agent 决定是否拆子 Agent）。
- 技术选型与关键架构决策有简短说明和理由。
- 第一个切片是"最薄可跑通"的骨架，能尽早验证范式跑得通。
- 定义本项目的**确定性验证命令**（编译 + 测试），供 lode-build 落成 `.lode/verify.sh`、由 Stop 门禁实跑——把"编译/测试过没过"从模型自评移交给程序。
- **播种任务看板**：切片定下后，把切片列表**写进原生 todo 列表**（每片一项），作为 build / auto 的实时进度看板起点——用户从界面就能看到「拆了几片」。

## 改现有代码时额外（每个切片必带）

读 `architecture.md` + spec 的 delta，给每个改动切片补：
- **爆炸半径**：这个切片会动哪些文件/模块、谁调用它（用 codegraph/调用关系定，别拍脑袋）。
- **回归面**：会牵动哪些现有行为；对应要跑的既有测试范围（喂给全量回归门禁）。
- **基线 / 钉行为**：动手前先**实跑现有测试存基线**；对要改且无测试的区域，先补 **characterization 测试**钉住当前行为，再改。
- **迁移策略**：涉及 schema/接口/数据格式变更，用 **expand → migrate → contract**（先兼容并存、再迁移、最后收口），别一刀切断。
- 允许**跨切片重构**：发现早先切片设计错了，专门开一个"重构切片"修，前提是回归网兜底——不是硬扛着错误设计往下堆。

## Guardrails（红线）

- 不要拆成一堆碎工具/碎函数任务 —— 给能力、给可验收切片，让 Builder 自己组织实现。
- 每个切片的完成标准必须可程序判断，不能是"差不多"。
- 计划服务于循环，不追求一次写死；允许 Build 过程中回来调整——**调整 = 生成新版 `.lode/plan/<功能简称>-<ts>.md`，不改旧文件**，让每次 replan 都留痕。
- 若已有设计原型代码，计划要明确**在其基础上直接开发、复用代码**，而非"仅作参考"重写。
- 计划先给用户拍板再进 `lode-build`。
