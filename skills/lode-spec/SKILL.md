# Product Spec Builder（需求收集）

主线第 ① 环，也是「毒舌」最集中的地方。通过结构化访谈把模糊想法**逼问**成一份能直接开发的 `Product-Spec.md`。

## Usage（什么时候用）

- 用户给了模糊想法，需求边界还没钉死。
- 进入 `lode-plan` 之前，先把"要做什么"问清。

## 怎么问（薄在步骤，厚在标准）

不写"第一步问什么第二步问什么"的脚本。真正写厚的是一份**问题库**（落地到 `.lode/<project>/question-bank.md`，起步模板见本 skill 的 `references/question-bank-spec.md`）：每个问题配「什么答案能接受、什么答案必须打回去」。模型按用户回答动态决定下一个问题，问题库只在跑偏时把它拽回来。把"怎么做"删掉，把"什么算好"写厚。

四个手法（逼问高效的关键）：

1. **选择题优先**：每个关键问题给 2–3 个**具体选项**让用户挑/否，而不是开放式问。
   例：「第一版更像 ① 快速清理工具 ② 内容重组 Agent ③ 生成式 Agent？」——选完再追问，比"你想要什么功能"快得多，也不挤牙膏。
2. **主动联网补域**：缺行业/领域知识时**自己上网搜**补齐，别反问用户。
3. **问题分流**：只问"用户才知道答案"的决策；属于实现细节、或 Skill 里可由用户自定义的，自己定或留到后面，**别拿这些烦用户**。
4. **边界探针**：主动抛"会不会无限膨胀"的边界问题，让用户早划线（如"手动能力做到哪算够？"）。

## 先亮假设（动手前必做）

逼问前，先把你对**核心决策**的关键假设一次性摆出来让用户一眼纠正——错误假设在后续自驱循环里会指数级放大：

```
我先按这些假设理解，错就现在打断我：
1. 这是 <平台/形态>，不是 <另一种>
2. 目标用户主要是 <…>
3. 这一版范围到 <…> 为止
→ 不纠正我就按此继续逼问。
```

## 棕地：用 delta 模式（改老项目时）

目标落在已有项目时，spec 不是"做什么"，而是**改什么**。先读 `lode-recon` 的 `System-Map.md`，然后按 delta 写：
- **现状**：现在的行为是什么（针对要改的那块）。
- **目标**：改完应该是什么。
- **绝不能破坏（不变量 / 回归面）**：哪些现有行为、数据、接口必须保持不变——这一栏直接决定 build 阶段要钉住的 characterization 测试和门禁要跑的回归范围。
- **受影响模块**：从 System-Map 标出会被牵动的地方（供 plan 做影响分析）。

## Done（什么算合格）

产出 `.lode/<project>/Product-Spec.md`，且满足：
- 价值主张 + 目标用户 + 核心场景说清楚。
- 功能需求分层（这一版做什么 / 先不做什么），每条可验收。
- 明确写出**范围边界**（不做什么），防止无限膨胀。
- 关键约束（平台、性能、隐私、离线/在线、产品形态）已定。
- 含用户故事、主流程、Agent 要用到的工具/能力、布局意向、外部依赖。
- **棕地额外**：现状→目标的 delta + **绝不能破坏清单**（不变量/回归面）+ 受影响模块。

## Guardrails（红线）

- **禁止阿谀奉承。** AI 天生顺着人说——你被夸爽了，需求还是一团浆糊。这里把规则写死：直白、追问到底、不接受模糊。
- 模糊处必须逼问，一次问准几个关键点；别替用户假设核心决策。
- 不写实现方案、不选技术栈（那是 Planner/Builder 的事）。
- 描述**能力**，不要把需求拆成一堆碎工具清单。
- 用户纠正了你的判断（如你建议保守、被顶回去），抓成 Signal 写进 `signals.jsonl`，交给自进化。
- 写完先给用户确认再进下一环。
