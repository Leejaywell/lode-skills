# Product Spec Builder（需求收集）

主线第 ① 环，也是「毒舌」最集中的地方。通过结构化访谈把模糊想法**追问**成一份能直接开发的 `docs/spec.md`。

## Usage（什么时候用）

- 用户给了模糊想法，需求边界还没钉死。
- 进入 `lode-plan` 之前，先把"要做什么"问清。
- **任何项目都从这里进**（从零新建、在自己项目上加功能、改外来代码——一个 `/lode-spec` 全包）。

## 开局：自动铺好该有的东西（用户零判断）

进项目先**自动**把 Lodestar 跑循环要的几样备齐，不让用户判断"什么时候装":

1. **规则 `CLAUDE.md`**：项目根没有就**先告知用户一句、再铺一份**——来源：插件装 `${CLAUDE_PLUGIN_ROOT}/CLAUDE.md`、脚本装 `~/.claude/lodestar/CLAUDE.md`（都找不到就问用户 Lodestar 装哪）。**项目根已有同名 `CLAUDE.md` 就绝不动它**——那很可能是用户自己项目的规则：不覆盖、不静默改；告知用户「检测到你已有 CLAUDE.md」，经同意再把 Lodestar 规则以 `<!-- LODESTAR:BEGIN/END -->` 包块**追加**进去（可整块移除），未同意就保持原样、本 session 靠技能+门禁兜底。**注意**：项目 `CLAUDE.md` 一般在会话开始时加载，中途铺的要**下个 session 才全程生效**；本 session 的元规则靠技能正文 + 门禁兜底——铺完跟用户说一句"规则已就位，新开会话后全程生效"。
2. **现状图 `system-map.md`**：见下表按情况备好。（`verify.sh` 不在这里铺——它由 `lode-build` 在开发真正开始那一刻用真实编译/测试命令自动写。）
3. **需求库 `docs/spec.md`**：已有就**先读进来、在其上增改**（绝不整份重写），没有才新建；并确保项目 `.gitignore` **忽略 `.lode/`、跟踪 `docs/spec*.md`**（spec 是进库的交付物，`.lode/` 是运行态不进库）。

> 这些都是**流程自动做**的,不是甩给用户的前置命令。想手动一次性铺,可选 `/lode-init`（一般用不到）。

### 现状图怎么备（`system-map.md` 是任何项目都该有的活地图）

`system-map.md` 不是"改现有代码才产出"的东西，而是**这个系统现在长什么样**的常驻记录。按当前情况把它备好（**AI 自己判断、自己做，不用用户选**）：

| 当前情况（看要碰的代码现在存不存在 + 有没有本项目 `.lode` 历史） | 怎么备现状图 |
|---|---|
| **还没代码**（从零新建第一个目标） | 起一个最小骨架；现状=空，delta 就是"从空到 X" |
| **有代码 + 本项目自己建的**（有 `.lode` 历史/上轮 changelog） | 现状已知，读现有 `system-map.md` 轻量刷新即可，**不重扫** |
| **有代码 + 外来/遗留**（无 `.lode` 历史） | 需要读代码补图：**库小**自己读；**库大/陌生**→ **派 `lode-recon` 子代理**（见 `agents/lode-recon.md`）用干净脑子去读、只把 `system-map.md` 带回来，别让海量代码污染本 session 的追问上下文 |

> 用户永远只敲一个 `/lode-spec`：现状图怎么来由本环自己定。**项目不是永久"新"或"旧"**——做完第一个目标有了代码，下一个目标自然落到"有代码"那档，这个滑动是自动的，与 `/lode-auto` 的检测一致。

## 怎么问（薄在步骤，厚在标准）

不写"第一步问什么第二步问什么"的脚本。真正写厚的是一份**问题库**（落地到 `.lode/question-bank-spec.md`，起步模板见本 skill 的 `references/question-bank-spec.md`）：每个问题配「什么答案能接受、什么答案必须打回去」。模型按用户回答动态决定下一个问题，问题库只在跑偏时把它拽回来。把"怎么做"删掉，把"什么算好"写厚。

四个手法（追问高效的关键）：

1. **选择题优先**：每个关键问题给 2–3 个**具体选项**让用户挑/否，而不是开放式问。
   例：「第一版更像 ① 快速清理工具 ② 内容重组 Agent ③ 生成式 Agent？」——选完再追问，比"你想要什么功能"快得多，也不挤牙膏。
2. **主动联网补域**：缺行业/领域知识时**自己上网搜**补齐，别反问用户。
3. **问题分流**：只问"用户才知道答案"的决策；属于实现细节、或 Skill 里可由用户自定义的，自己定或留到后面，**别拿这些烦用户**。
4. **边界探针**：主动抛"会不会无限膨胀"的边界问题，让用户早划线（如"手动能力做到哪算够？"）。

## 遇到 UI / 布局：用线框对齐，别用文字让用户猜

碰到界面结构、布局、屏与屏关系这类**说不清**的点，先判一句——**用户「看到」比「读到」更好懂吗？** 是才上视觉，按轻到重三级：

- **概念题**（要什么风格、这功能干嘛）→ 照常文字选择题，别上图。
- **结构/位置题**（几栏、什么放哪、信息层级）→ **直接画 ASCII 粗线框**对齐，零文件、最快：
  ```
  ┌─侧栏─┬───预览区───┐
  │ 列表 │  大图+时间轴 │   选这种「左列表 + 右预览」，还是上下分栏？
  └──────┴────────────┘
  ```
- **要对比/带点真实感才说得清** → 写一个低保真 `.lode/wireframe.html`（纯静态、可双击），给一句 `open .lode/wireframe.html`，看完在终端说。
- **要真·高保真 / 可点击原型** → 这是信号，别在 spec 硬画：转 `/lode-brief` → `/lode-design`（那里才出可直接打开的成品原型）。

> 第一次要画图前，先单独说一句"我画个线框给你看？"征得同意，之后**逐点**判「看 vs 读」（不是每个 UI 话题都上图）。谈定的布局意图记进 `docs/spec.md` 的「布局意向」（可附线框）。**spec 只为消歧义，不出成品设计。**

## 先亮假设（动手前必做）

追问前，先把你对**核心决策**的关键假设一次性摆出来让用户一眼纠正——错误假设在后续自驱循环里会指数级放大：

```
我先按这些假设理解，错就现在打断我：
1. 这是 <平台/形态>，不是 <另一种>
2. 目标用户主要是 <…>
3. 这一版范围到 <…> 为止
→ 不纠正我就按此继续追问。
```

## 改现有代码：用 delta 模式（改已有项目时）

目标落在已有项目时，spec 不是"做什么"，而是**改什么**。现状以开局补好的 `system-map.md` 为准，按 delta 写：
- **现状**：现在的行为是什么（针对要改的那块）。
- **目标**：改完应该是什么。
- **绝不能破坏（不变量 / 回归面）**：哪些现有行为、数据、接口必须保持不变——这一栏直接决定 build 阶段要钉住的 characterization 测试和门禁要跑的回归范围。
- **受影响模块**：从 system-map 标出会被牵动的地方（供 plan 做影响分析）。

## Done（什么算合格）

产出 `docs/spec.md`，且满足：
- 价值主张 + 目标用户 + 核心场景说清楚。
- 功能需求分层（这一版做什么 / 先不做什么），每条可验收。
- 明确写出**范围边界**（不做什么），防止无限膨胀。
- 关键约束（平台、性能、隐私、离线/在线、产品形态）已定。
- 含用户故事、主流程、Agent 要用到的工具/能力、布局意向、外部依赖。
- **改现有代码时额外**：现状→目标的 delta + **绝不能破坏清单**（不变量/回归面）+ 受影响模块。

## Guardrails（红线）

- **禁止阿谀奉承。** AI 天生顺着人说——你被夸爽了，需求还是一团浆糊。这里把规则写死：直白、追问到底、不接受模糊。
- 模糊处必须追问，一次问准几个关键点；别替用户假设核心决策。
- **无现状图不进追问**：要改的代码已存在却没有 `system-map.md`，先把图备好（小库自己读、大库派 `lode-recon` 子代理）再写 delta，不许凭想象编现状。
- 不写实现方案、不选技术栈（那是 Planner/Builder 的事）。
- UI/布局说不清就画粗线框对齐意图，**不在 spec 出成品设计**（成品归 brief/design）。
- 描述**能力**，不要把需求拆成一堆碎工具清单。
- 用户纠正了你的判断（如你建议保守、被顶回去），抓成 Signal 写进 `signals.jsonl`，交给自进化。
- **先读后改、只演进不堆叠**：开局先读已有 `docs/spec.md` 在其上增改；被取代的条目移入文末「已调整/已废弃」归档，**不整份重写、不堆增量**。每次实质变更追加一行进 `docs/spec-changelog.md`（日期/改了什么/为什么），完整历史交给 git。
- 写完先给用户确认再进下一环。

## → 下一步
需求定了,二选一:
- 想一步步盯 → `/lode-plan` 拆切片(界面是成败关键可先 `/lode-brief` 做设计)。
- 想 agent 自动跑完剩下的 → `/lode-auto`(它会**直接拿这份 `docs/spec.md` 当输入**拆计划、逐切片 build 到底,不重新问需求)。
