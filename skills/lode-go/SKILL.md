# Go Creator（写 Go）

扩展技能。**Go 是整个循环的入口**：前面定的所有标准和规则，最后都靠发一条 Go 交给 AI 跑，AI 进入循环直到达成 Go 为止。

> 为什么要专门的 Skill？因为大部分人写不好 Go——目标含糊、标准没法验证，结果要么 AI 跑偏，要么它没干完却以为干完了。**写一条好 Go 比写代码还难。** 而最会写 Go 的是 AI 自己：它在现场，知道聊到哪、项目什么状态、接下来该干什么。让它结合上下文替你把 Go 写好，你只需扫一眼点发送。

## Usage（什么时候用）

- 开发计划已就绪，要把一个 Face（或全部 Face）交给 AI 自驱执行。
- 任何环节缺一条可执行、可验收的 Go。

## Done（一条 Go 包含五段）

1. **目标**：要完成什么（一句话交付物，如"完成 dev-plan Phase 1"）。
2. **完成标准**：什么算完成——可程序判断（编译零报错 / 测试全过 / 审查通过），逐条列。
3. **验收方式**：如何验证——可检查的证据（关键命令输出 / 测试结果 / 新建修改的文件清单 / 审查报告）。
4. **约束**：不许碰什么（如不改 product-spec/design-brief、不动已定 UI、**不 push、不删原型文件**，除非用户确认）。注意：每个 Face 审过后的**本地 commit 是允许的**（作回滚点），约束的是对外不可逆动作（push/上架）。
5. **执行策略 + 熔断**：目标导向，一条路走不通换多种方法都试过才停；长任务持续推进——**但设熔断线：同一 Face 连续失败 ≥3 次、或超出 token 预算，就停下找人，不无限重试**。

## 先亮假设（生成 Go 前必做）

把 Go 交给 AI 自驱前，先列出你对**目标/范围/验收**的关键假设让用户一眼确认——自驱循环里错误假设会指数级放大、跑很久才暴露：

```
这条 Go 我按这些假设来定，错就现在改：
1. 目标范围到 <Face N / Phase N> 为止
2. <技术/数据/依赖> 维持现状不变
3. 验收以 <verify.sh 全绿 + 审查通过> 为准
→ 不纠正就发送执行。
```

## 三种执行方式（配合 lode-plan / lode-build）

1. 主 Agent 直接用 `lode-build` 跑完整个计划。
2. 把**第一个 Face** 写成 Go，复制执行，循环推进（最常用）。
3. 按**全部 Face 整体规划**写**一条 Go** 一次性开发完（熟练后最高效）。

## Guardrails（红线）

- 完成标准不能是"差不多/看着行"；不可判断 = 没有标准。
- Go 定终点，不写实现步骤。
- 弹窗/草稿写完通常无需二次编辑，除非有特殊目标要加。
