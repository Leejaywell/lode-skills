# Design Brief Builder（设计文档）

主线第 ② 环。像个设计师在采访甲方：把用户脑子里模糊的"感觉/调性"，通过访谈翻译成**具体、可执行的设计决策**。

## Usage（什么时候用）

- `docs/spec.md` 已确认，进入设计阶段（本环可选）。
- 用户对外观有期待但只能用形容词描述（"高级一点""干净""有科技感"）。
- 需要在动手做高保真设计稿（`lode-design`）或写前端前，先定规范。

## 怎么问（把"感觉"敲定成具体决策）

靠一份问题库（起步模板见本 skill 的 `references/question-bank-design.md`），逐项用**选择题**收敛：

1. **组合参考 + 给推荐**：每个方向给 2–3 个**具体产品组合**让用户挑，并给出你的推荐。
   例：「① 剪映结构 + Linear 气质　② CapCut + Cursor 工程感　③ 达芬奇 + 重控制台感——我建议 ①」。
   逐项过：第一印象参考 / 主题色（深·浅）/ 信息密度 / 圆角层级 / 字体 / Agent 状态呈现 / 图标 / 列表 vs 网格 / 预览区 / 空态 / 文案语气。
2. **反参考（必问一题）**：「什么**绝对不能出现**？你**讨厌**哪些产品的感觉？」——反参考比正参考更能锁定方向。
3. **被否就换一批**：用户说"太 AI 了 / 不对"，立刻换一组新选项（如配色 冷蓝→琥珀黄/暖白/珊瑚红），迭代收敛而不是卡住。
4. **借力已装技能**：可调用 `high-end-visual-design` / `frontend-design` / `design-taste-frontend` 和 ECC `web/design-quality` 规则，避免模板感。

## Done（什么算合格）

产出设计规范，落地到 `.lode/design/<功能简称>-<YYYY-MM-DD_HH_MM_SS>.md`（**每次重出方向生成新文件、绝不覆盖旧版**，保留设计演进史；时间戳到秒防撞名。取最新：`ls -t .lode/design/*.md | head -1`），且满足：
- 选定一个**明确的设计方向**（编辑杂志风/新拟态/玻璃拟态/Swiss…），不用"clean minimal"之类空话。
- **参考与反参考**都写明（要什么 + 绝不要什么）。
- 可落地的设计变量（design token，即配色/字号/间距等命名变量，**与 LLM 的 token 无关**）：配色（含语义色）、字体搭配、间距/圆角/阴影节奏。
- 关键状态：hover / focus / active / 禁用 / 空态 / 加载 / 错误。
- 无障碍与响应式底线（对比度、reduced-motion、断点）。
- 信息架构 + 页面清单（含页面关系）；每条决策能回溯到 spec 的用户与场景。

## Guardrails（红线）

- 拒绝模板感：不默认 dark mode，不要千篇一律的卡片网格 + 居中标题 + 渐变球。
- 把"感觉"落成可判断的规范，别留"看着办"。
- **新方向另存新版、不覆盖**：改设计方向时**生成新 `.lode/design/<功能简称>-<ts>.md`**（当前决策一份写清、内部不堆叠旧版），旧文件原样留痕；下游永远读最新。影响需求的决策回写 `docs/spec.md` + `docs/spec-changelog.md` 记一行。
- 不在此环写代码或生成完整页面（那是 lode-design / lode-build）。
- 方向定下来先和用户确认再扩展。
