# Skill Builder（造新技能）

扩展技能。把一项能力沉淀成一个新的 `SKILL.md`。范式里专门用来"造技能"的技能。

## Usage（什么时候用）

- 需要给系统（或正在开发的 Agent 产品）加一项新能力。
- 现有 Skill 写法不对（写成了操作手册、或拆得太碎），要重写。

## Done（什么算合格）

产出一个符合范式的 `SKILL.md`：
- frontmatter 有清晰的 `name` 和 `description`（description 决定何时被触发，要写准）。
- 正文只写**三类情况**：
  - **Usage**：什么时候用。
  - **Done**：做到什么算合格。
  - **Guardrails**：什么红线不能碰。
- "具体怎么做"留给模型自己组织；只在需要把"什么算好"写厚的地方（如问题库）才写厚。
- 较大的能力把参考材料放进 skill 自己的 `references/`，正文只留指针（progressive disclosure）——别把外部相对路径写进正文，装机后会断链。

## 发布前最小自测（落地版"踩坑再改"）

写完草稿别直接收——先跑一遍最小评测，这是「真实失败再立规则」的**发布前**版本：

1. 造 **2–3 个真实触发 prompt**（用户真会那么说的话），交用户一眼确认。
2. 各跑一遍，看两件事：**触发是否准**（该触发时触发、不抢别的 skill），**产出是否达标**（对照本 skill 的 Done）。
3. 没过就改 `description`（触发问题）或正文标准（产出问题），重跑，直到稳定。
4. 触发不准最常见的根因是 `description` 不够具体或与已装 skill 撞词——优先调它。

> 产出客观可验证的 skill（文件变换/数据抽取/固定流程）值得这步；纯主观风格类（文案/审美）可跳过。

## Guardrails（红线）

- **少造工具，多给能力。** 不要把一项能力拆成一小堆专用小工具（找字幕一个、切片段一个…），那样反而笨。给完整的通用能力，让模型自己组合——**聪明是放出来的，不是设计出来的**。
- 不写分步操作手册（写得越细越压模型上限）。
- 能让程序判断的规则，做成 hook 门禁，别写进 Skill 让模型反复念。
- 新技能先给用户确认定位与边界，再落地。
